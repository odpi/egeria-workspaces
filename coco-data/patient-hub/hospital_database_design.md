# Hospital patient database — design document

## Overview

This document describes the design of the hospital patient database, the purpose of each table, and the reasoning behind individual field choices. The database is implemented in PostgreSQL and is intended to support a clinical environment where patient safety, data integrity, and a complete audit trail are primary concerns.

The schema is organised around the **visit** as the central unit of clinical activity. Every piece of clinical data — treatments, measurements, lab results, and medical images — is attached to a specific visit rather than directly to a patient. This reflects how hospitals actually work: a blood pressure reading or an X-ray belongs to a particular attendance, not abstractly to a person. It also makes it straightforward to answer questions like "what happened during this admission?" with a single join.

---

## Design principles

**UUIDs as primary keys.** Every table uses `UUID` as its primary key, generated by `gen_random_uuid()`. UUIDs are safe to generate outside the database (useful for pre-creating IDs in application code before the insert), they do not leak information about record counts, and they avoid the merge problems that arise with sequential integers when data is imported from multiple sources.

**Timestamps with time zone.** All date-time columns use `TIMESTAMPTZ` (timestamp with time zone) rather than plain `TIMESTAMP`. This ensures that times are stored in UTC and converted correctly regardless of the server's local time setting — important in a clinical record where the precise time of a measurement or treatment can matter medically and legally.

**Flexible type/value/unit pattern.** Rather than creating a fixed column for every possible measurement or lab analyte (which would require a schema change every time a new test is introduced), the `measurement` and `lab_result` tables use a `type` or `analyte` column alongside `value` and `unit`. This is a proven approach in clinical informatics and accommodates an open-ended set of observations without structural change.

**Audit at the database level.** The audit log is implemented as a PostgreSQL trigger rather than relying on application code. This means that no write to the database — regardless of which application, tool, or direct connection makes it — can bypass the audit trail. Every insert, update, and delete is captured automatically.

## Tables

### `department`

Represents a hospital department or clinical unit (e.g. Cardiology, Emergency, Radiology).

| Field | Type | Reasoning |
|---|---|---|
| `department_id` | UUID PK | Stable identifier independent of naming changes. |
| `name` | VARCHAR(120) | Full display name of the department. |
| `code` | VARCHAR(20) UNIQUE | Short code used in reporting and HL7 messages (e.g. `CARD`, `ED`). Unique constraint prevents accidental duplicates. |
| `location` | VARCHAR(120) | Physical location within the hospital (building, floor, ward). |
| `head_clinician_id` | UUID FK → clinician | The current clinical lead. Optional (NULL while the post is vacant). Uses a deferred foreign key because `clinician` is created after `department`. |
| `created_by` | UUID FK → clinician | Who created this record. Part of the standard audit set present on every table. |
| `created_at` | TIMESTAMPTZ | When the record was created. Set to `now()` by default; the application should not override this. |
| `updated_by` | UUID FK → clinician | Who last modified the record. Set by the application via the session variable `app.current_clinician_id`. |
| `updated_at` | TIMESTAMPTZ | Automatically stamped by the `fn_set_updated_at()` trigger on every `UPDATE`; the application never needs to set this. |

---

### `clinician`

Any member of staff who interacts with patient data — doctors, nurses, radiographers, pharmacists, and administrative staff with system access. This table acts as the identity anchor for all audit columns across the schema.

| Field | Type | Reasoning |
|---|---|---|
| `clinician_id` | UUID PK | Stable identifier. |
| `first_name`, `last_name` | VARCHAR(80) | Display and search purposes. Separated rather than combined to allow proper sorting and addressing. |
| `role` | VARCHAR(80) | Job title or clinical role (e.g. `Consultant`, `Registrar`, `Staff Nurse`). Free text to avoid a rigid hierarchy that would need constant maintenance. |
| `department_id` | UUID FK → department | The clinician's home department. A clinician can perform treatments in other departments (captured on the `treatment` row), but has a primary affiliation here. |
| `gmc_number` | VARCHAR(20) UNIQUE | GMC registration number for doctors; NMC pin for nurses. Optional (NULL for non-registered roles). The unique constraint prevents the same registration being entered twice. |
| `created_by`, `created_at`, `updated_by`, `updated_at` | — | Standard audit columns (see above). `created_by` is a self-referential FK — a clinician record is always created by another clinician (typically an administrator). |

---

### `patient`

The core demographic record for each patient. This table deliberately holds only information that belongs to the patient as a person — not clinical observations, which belong on visits.

| Field | Type | Reasoning |
|---|---|---|
| `patient_id` | UUID PK | Internal stable identifier. |
| `nhs_number` | CHAR(10) UNIQUE | The NHS number is the national patient identifier in England and Wales. `CHAR(10)` enforces the fixed length. The unique constraint prevents duplicate registrations. |
| `first_name`, `last_name` | VARCHAR(80) | Separated for sorting, salutation, and search. |
| `date_of_birth` | DATE | Used for patient matching, age calculations, and eligibility checks. |
| `sex` | VARCHAR(20) | Uses a short VARCHAR rather than a boolean or enum to accommodate evolving terminology (`Male`, `Female`, `Non-binary`, `Not specified`, etc.) without requiring schema changes. |
| `address`, `postcode` | TEXT / VARCHAR(10) | Address is free text given the variety of international formats. Postcode is kept separate for geographic analysis and NHS reporting. |
| `phone`, `email` | VARCHAR | Contact details for appointment notifications. |
| `emergency_contact` | TEXT | Free text for name, relationship, and phone number of the person to contact in an emergency. Free text is appropriate here — rigid structure would add complexity without benefit. |
| `blood_type` | VARCHAR(5) | Stored at the patient level (e.g. `A+`, `O-`) because it is a stable biological characteristic, not a per-visit finding. |
| `allergies` | TEXT | Free text summary of known allergies. A structured allergy table could be added in a later iteration if clinical decision support is required. |
| `created_by`, `created_at`, `updated_by`, `updated_at` | — | Standard audit columns. |

**Indexes:** `nhs_number` (lookup by NHS number), composite `(last_name, first_name)` (name search), `date_of_birth` (DOB-based matching).

---

### `visit`

A single attendance at the hospital, whether inpatient, outpatient, day case, or emergency. The visit is the hub to which all clinical activity is attached.

| Field | Type | Reasoning |
|---|---|---|
| `visit_id` | UUID PK | Stable identifier. |
| `patient_id` | UUID FK → patient | Which patient attended. |
| `department_id` | UUID FK → department | The primary receiving department (e.g. Emergency, Cardiology). A patient may receive treatment from other departments during the same visit; those departments are recorded on the `treatment` rows. |
| `attending_clinician_id` | UUID FK → clinician | The responsible clinician for the visit — the consultant of record or attending physician. |
| `admitted_at` | TIMESTAMPTZ | Date and time of arrival or admission. Defaults to `now()` if not explicitly provided. |
| `discharged_at` | TIMESTAMPTZ | NULL while the patient is still admitted. A NULL discharge date is the operational definition of an active inpatient. |
| `visit_type` | VARCHAR(40) | Categorises the attendance: `Inpatient`, `Outpatient`, `Emergency`, `Day case`. Drives reporting and capacity management. |
| `reason` | TEXT | The presenting complaint or referral reason in clinical language. |
| `discharge_notes` | TEXT | Summary written at discharge — diagnoses, outcomes, follow-up instructions. NULL until discharge. |
| `created_by`, `created_at`, `updated_by`, `updated_at` | — | Standard audit columns. |

**Indexes:** `patient_id` (all visits for a patient), `department_id` (all visits to a department), composite `(admitted_at, discharged_at)` (date-range queries and active admission checks).

---

### `treatment`

A specific clinical intervention carried out during a visit — medication administration, a surgical procedure, a physiotherapy session, a dressing change, and so on. There can be many treatments per visit.

| Field | Type | Reasoning |
|---|---|---|
| `treatment_id` | UUID PK | Stable identifier. |
| `visit_id` | UUID FK → visit | The visit during which this treatment occurred. |
| `department_id` | UUID FK → department | The department that performed the treatment, which may differ from the visit's primary department (e.g. a Cardiology patient sent to Radiology for an intervention). |
| `clinician_id` | UUID FK → clinician | The clinician who administered the treatment. |
| `treatment_type` | VARCHAR(80) | High-level category: `Medication`, `Surgery`, `Physiotherapy`, `Wound care`, `Procedure`, etc. Used for filtering and reporting. |
| `description` | VARCHAR(255) | Concise description of the specific treatment (e.g. `Amoxicillin 500mg PO`, `Appendectomy`). |
| `notes` | TEXT | Extended clinical notes, observations, or complications. |
| `performed_at` | TIMESTAMPTZ | When the treatment was carried out. Important for medication timing, pre/post-op sequencing, and medico-legal records. |
| `created_by`, `created_at`, `updated_by`, `updated_at` | — | Standard audit columns. |

---

### `measurement`

Point-in-time clinical observations: vital signs, anthropometric measurements, and other numerical readings. The flexible `type` / `value` / `unit` model means new measurement kinds can be introduced without any schema changes.

| Field | Type | Reasoning |
|---|---|---|
| `measurement_id` | UUID PK | Stable identifier. |
| `visit_id` | UUID FK → visit | The visit during which this measurement was taken. |
| `clinician_id` | UUID FK → clinician | Who took the measurement. |
| `measured_at` | TIMESTAMPTZ | When the reading was taken — critical for trending vital signs over a stay. |
| `type` | VARCHAR(80) | The kind of measurement: `Height`, `Weight`, `Systolic BP`, `Diastolic BP`, `Heart rate`, `SpO2`, `Temperature`, etc. No enum is used deliberately so new types can be introduced at the application layer without a schema migration. |
| `value` | NUMERIC(10,4) | The numerical reading. `NUMERIC` (exact decimal) is used rather than `FLOAT` to avoid floating-point rounding errors in clinical values. |
| `unit` | VARCHAR(20) | The unit of measurement: `cm`, `kg`, `mmHg`, `bpm`, `%`, `°C`, etc. Stored alongside the value to prevent misinterpretation. |
| `notes` | TEXT | Any qualifying notes (e.g. `Patient standing`, `Left arm`, `Post-exercise`). |
| `created_by`, `created_at`, `updated_by`, `updated_at` | — | Standard audit columns. |

**Indexes:** `visit_id`, `type` (retrieve all readings of a specific type across visits for trending).

---

### `lab_result`

Results from laboratory analysis of samples — blood panels, urinalysis, microbiology cultures, histology and biopsy pathology. One row per analyte, grouped into a panel by sharing `sample_type`, `collected_at`, and `panel`.

| Field | Type | Reasoning |
|---|---|---|
| `result_id` | UUID PK | Stable identifier. |
| `visit_id` | UUID FK → visit | The visit that generated this result. |
| `clinician_id` | UUID FK → clinician | The ordering clinician. |
| `sample_type` | VARCHAR(40) | The biological sample: `Blood`, `Urine`, `Biopsy`, `CSF`, `Swab`, etc. |
| `collected_at` | TIMESTAMPTZ | When the sample was collected — distinct from when the result was reported. The gap between these two timestamps indicates laboratory turnaround time. |
| `resulted_at` | TIMESTAMPTZ | When the result was validated and released by the laboratory. NULL until the result is available. |
| `panel` | VARCHAR(80) | The test panel: `FBC` (full blood count), `U&E` (urea and electrolytes), `LFT` (liver function), `Histology`, etc. Allows grouping of related analytes. |
| `analyte` | VARCHAR(80) | The specific substance measured: `Haemoglobin`, `Sodium`, `Creatinine`, `ALT`, etc. |
| `value` | NUMERIC(12,4) | The numerical result. NULL for qualitative results (use `value_text`). `NUMERIC` avoids floating-point errors. |
| `value_text` | VARCHAR(120) | Qualitative or semi-quantitative results: `Positive`, `Negative`, `No growth`, `Scanty growth`, `+++`, etc. |
| `unit` | VARCHAR(20) | Unit of measurement (e.g. `g/L`, `mmol/L`, `IU/L`). NULL for qualitative results. |
| `reference_range` | VARCHAR(40) | The normal range for this analyte in the reporting laboratory (e.g. `130–170`, `3.5–5.0`). Stored as a string because ranges vary by age, sex, and laboratory method. |
| `flag` | VARCHAR(10) | Abnormality indicator: `H` (high), `L` (low), `HH` (critically high), `LL` (critically low), `A` (abnormal — for qualitative results). Standard HL7 coding. |
| `created_by`, `created_at`, `updated_by`, `updated_at` | — | Standard audit columns. |

**Indexes:** `visit_id`, `analyte` (trend a specific analyte over time across visits).

### `medical_image`

Metadata and file references for medical imaging: X-rays, CT scans, MRI, ultrasound, nuclear medicine, and any other modality. The image files themselves are stored on a PACS server, network storage, or object store — not in the database. This table holds the metadata needed to locate, identify, and contextualise each image.

#### Core image fields

| Field | Type | Reasoning |
|---|---|---|
| `image_id` | UUID PK | Stable internal identifier. |
| `visit_id` | UUID FK → visit | The visit during which imaging was performed. |
| `clinician_id` | UUID FK → clinician | The ordering clinician. |
| `modality` | VARCHAR(20) | DICOM modality code: `CR` (computed radiography), `DX` (digital X-ray), `CT`, `MR`, `US` (ultrasound), `NM` (nuclear medicine), `PT` (PET). Using standard DICOM codes ensures interoperability with PACS systems. |
| `body_part` | VARCHAR(80) | Plain-text description of the anatomical region (e.g. `Chest`, `Left knee`, `Abdomen`). |
| `captured_at` | TIMESTAMPTZ | When the image was acquired. |
| `file_path` | TEXT | The path or URI to the image file. This may be an absolute filesystem path, a relative path, or a URI such as `s3://hospital-images/2026/01/...`. Using TEXT allows any addressing scheme without constraint. |
| `file_format` | VARCHAR(20) | `DICOM` for native DICOM files; `JPEG`, `PNG`, or `TIFF` for derived or scanned images. |
| `file_size_bytes` | BIGINT | Used for storage accounting and to detect incomplete transfers. |

#### DICOM identifier fields

DICOM (Digital Imaging and Communications in Medicine) is the universal standard for medical imaging. Every DICOM object carries a hierarchy of globally unique identifiers in its header. Storing these in the database enables cross-referencing with PACS systems, radiology information systems (RIS), and HL7/FHIR interfaces.

| Field | DICOM tag | Reasoning |
|---|---|---|
| `dicom_study_uid` | StudyInstanceUID | Identifies the study — a set of image series acquired at one session for one patient. All series from the same imaging appointment share this UID. Indexed to allow retrieval of all images in a study. |
| `dicom_series_uid` | SeriesInstanceUID | Identifies a series within a study — typically one acquisition run (e.g. one CT phase). Multiple images from the same acquisition share this UID. Indexed for series-level retrieval. |
| `dicom_sop_uid` | SOPInstanceUID | Identifies a single DICOM object (one image, one SR document, etc.). Globally unique by construction. Carries a `UNIQUE` constraint: if the same DICOM object is received twice (common in PACS integration), the second insert is rejected rather than creating a duplicate. |
| `dicom_patient_id` | PatientID | The patient identifier embedded in the DICOM header by the imaging device. This may be the NHS number, a local hospital number, or a scanner-assigned value. It is stored separately from `nhs_number` because the two may differ, particularly for images imported from other institutions. |
| `dicom_accession_no` | AccessionNumber | The order number assigned by the radiology information system (RIS) when the imaging request was placed. Links the image back to the original order. Indexed to support worklist lookups. |
| `dicom_institution` | InstitutionName | The name of the institution whose equipment produced the image. Important in multi-site deployments or when images are received from external hospitals. |

#### Report fields

| Field | Type | Reasoning |
|---|---|---|
| `report` | TEXT | The radiologist's written report. Stored as free text; structured reporting could be added as a separate table if SNOMED-coded findings are required. |
| `report_by` | VARCHAR(120) | Name and GMC number of the reporting radiologist. Stored as text rather than a FK to `clinician` because the reporting radiologist may be from an external teleradiology service and not registered in the local clinician table. |

### `audit_log`

A complete, immutable record of every data change across all clinical tables. One row is written automatically for every `INSERT`, `UPDATE`, and `DELETE` on all eight clinical tables via the `fn_audit_trigger()` trigger function. Application code never writes to this table directly.

| Field | Type | Reasoning |
|---|---|---|
| `log_id` | BIGSERIAL PK | An auto-incrementing integer rather than UUID. This guarantees strict chronological ordering of audit entries, which is important when reconstructing the history of a record. A UUID would be unordered. |
| `table_name` | VARCHAR(60) | Which table was changed. Allows filtering audit history by entity type. |
| `record_id` | UUID | The primary key of the affected row. Allows filtering all changes to a single record. |
| `operation` | CHAR(6) | `INSERT`, `UPDATE`, or `DELETE`. A `CHECK` constraint enforces only these three values. |
| `performed_by` | UUID FK → clinician | The clinician whose session made the change. Sourced from the session-local variable `app.current_clinician_id`, which the application sets at the start of every transaction. If not set (e.g. a direct DBA connection), this is NULL — which is itself a useful audit signal. |
| `performed_at` | TIMESTAMPTZ | The exact timestamp of the change, set to `now()` by the trigger. |
| `old_data` | JSONB | The complete row as it was before the change. NULL on `INSERT`. Stored as JSONB so that the history is self-contained and readable regardless of future schema changes. |
| `new_data` | JSONB | The complete row after the change. NULL on `DELETE`. |
| `ip_address` | INET | The IP address of the client that made the change, sourced from `app.ip_address`. PostgreSQL's `INET` type validates the format and supports range queries. Useful for security investigations. |
| `application` | VARCHAR(80) | The name of the application or integration that wrote the data (e.g. `EPR`, `HL7-Gateway`, `API`, `PACS-Ingest`). Makes it possible to distinguish human-originated changes from automated interface activity. |

**Indexes:** `table_name`, `record_id`, `performed_by`, `performed_at`. These four cover the most common audit query patterns: all changes to a given record, all changes by a given clinician, and time-bounded audit reports.

#### Why triggers rather than application-level logging

If audit logging is implemented in application code, it can be bypassed — intentionally or accidentally — by direct database connections, migration scripts, bulk imports, or bugs. A database-level trigger fires unconditionally for every data modification regardless of its origin. This is the only reliable mechanism for a complete audit trail in a clinical environment.

#### How the trigger resolves the acting clinician

The trigger function reads three PostgreSQL session-local settings that the application sets at the start of every transaction:

```sql
BEGIN;
  SET LOCAL app.current_clinician_id = 'uuid-here';
  SET LOCAL app.ip_address           = '10.0.1.42';
  SET LOCAL app.application_name     = 'EPR';
  -- DML statements
COMMIT;
```

`SET LOCAL` scopes the values to the current transaction; they are automatically cleared on `COMMIT` or `ROLLBACK`. There is no risk of one session's identity persisting into the next transaction, even in a connection pool.

## Trigger functions

### `fn_set_updated_at()`

A `BEFORE UPDATE` trigger on every clinical table that sets `updated_at = now()`. This removes the responsibility from application code — the column cannot be omitted or set to the wrong value, because the database sets it unconditionally.

### `fn_audit_trigger()`

An `AFTER INSERT OR UPDATE OR DELETE` trigger on every clinical table. It reads the old and new row states, resolves the actor context from session variables, and inserts a row into `audit_log`. Because it is an `AFTER` trigger it runs after the DML succeeds — no audit row is written for a transaction that is rolled back.

## Conventions applied throughout

**`created_by` / `created_at` / `updated_by` / `updated_at`** appear on every clinical table. `created_at` and `updated_at` are set automatically by default values and triggers respectively. `created_by` and `updated_by` are set by the application via the session variable `app.current_clinician_id`.

**`NOT NULL` discipline.** Fields that are logically required carry `NOT NULL`. Fields that are genuinely optional (e.g. `discharged_at`, `email`, `dicom_institution`) are nullable, with NULL carrying a specific meaning (patient still admitted; email not provided; non-DICOM image).

**`NUMERIC` for clinical values.** All numeric measurement and lab result columns use `NUMERIC` (exact decimal arithmetic) rather than `FLOAT` or `REAL`. Floating-point types can introduce rounding errors; a haemoglobin of 13.0 g/L should never be stored as 12.999999998.

**VARCHAR lengths.** Column lengths are set generously but not arbitrarily. `VARCHAR(80)` for names is sufficient for virtually all real-world values; `VARCHAR(255)` for descriptions allows a full sentence. `TEXT` is used where there is no natural upper bound (free-text notes, file paths, reports).

**DICOM UID length.** DICOM UIDs follow a defined syntax and have a maximum length of 64 characters. All DICOM UID columns are therefore `VARCHAR(64)`.
